# Colors

This system is built using two layers:

1. Semantic tokens → used in UI
2. Primitives (scales) → raw color values

---

## Actions vs States

This system defines three distinct roles:

### Action variants

Define what the user can do.

| Variant     | Usage                            |
| ----------- | -------------------------------- |
| primary     | Main action on the screen        |
| accent      | Secondary but important action   |
| ghost       | Low emphasis action              |
| destructive | Dangerous or irreversible action |

---

### State colors

Define how the system responds.

| State   | Usage                            |
| ------- | -------------------------------- |
| success | Operation completed successfully |
| warning | Attention required               |
| error   | Operation failed                 |

---

### Neutral tokens

Define the visual foundation of the interface.

| Category               | Usage                                     |
| ---------------------- | ----------------------------------------- |
| backgrounds & surfaces | App background, cards, panels, containers |
| text                   | Primary, secondary, and muted text        |
| borders                | Dividers, outlines, and separators        |

Neutral tokens do not communicate action or feedback.
They provide structure, hierarchy, and readability.

---

### Core rule

If the user clicks it → it's an action variant
If the system shows feedback → it's a state
If it builds the interface structure → it's neutral

---

### Guidelines

- Do not use state colors as action variants
- Do not use accent for feedback
- Use destructive only for irreversible actions
- Use neutral tokens for layout, text, and borders
- Keep hierarchy clear and predictable
